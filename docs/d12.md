# DELI 12 - ProcFS Integ

## Objective
- started off tryna implement `even_fork` then realised the other things i had to do for that and then it became proc fs integ, yippee

## Learnings
- very big section here today, spent 7 hrs on all of this from 16:06 to rn 22:50, was very fun tho, a lot of types of things were done today, majority was prolly memory handling
- started off as even_fork imple, realized i need a proc node construction from pid, with metadata from procfs
- added ser_prfs.c and ser_prfs.h, for all proc fs related work
- so right now the nomenclature is
	- `proc_...` = in memory process objects
	- `even_...` = react to kernel events
	- `prfs_...` = the /proc fs
- new constructor fn, `proc *proc_load(pid)` 
	- allocates new proc
	- assigns pid
	- calls `prfs_look(proc *p` to flesh out metadata from procfs
	- returns  full `proc *p`
- all metadata is directly pushed onto `proc *p` no middlemen, one source of truth
- `strtol` is very strong, auto skips whitespaces, remembers where it left off, more convenient than `atoi`
- now the prfs stuff
	- `prfs_look()` 
		- orchestrator, `stat -> status -> fds`
	- `prfs_stat()` 
		- parses `/proc/<pid>/stat` extracts name, state and ppid
		- classic proc name parentheses handling issue solved by using `strchr` and `strrchr` to find and isolate the brackets and get name, the rest with `strtol`
	- `prfs_status()`
		- parses `/proc/<pid>/status` gets tgid, uid, gid (real and executive), and fdsize
		- later realized fdsize isnt like the number of fd's but rather the size of the file descriptor table, all good tho
	- `prfs_fd()`
		- parses `/proc/<pid>/fd` the reads each link, and gets full path
		- uses `opendir`, `readdir` and `readlink` to resolve all filepaths
- all of this had a lot of memory management omg
	- `proc_dest()` helper for destroying a proc and freeing all allocated memory and nulling any dangling pointers
- implemented the `even_fork` fn 
	- checks parent then child then attaches them
    - checks if parent exists
    - reconstructs missing parent from ProcFS
    - constructs child from ProcFS
    - attaches child into the process tree
    - inserts processes into the hash
    - also handles rollbacks
- observed real time short lifespan processes and what happens to them, ex. cat
	- they appear in the proc conn event stream as a fork
	- but theyre too short lived to extract the metadata
	- so some fork's are blank in the output

## Outcome
have a solid proc fs abstraction now, with clear memory management, and integ with proc conn directly to see fork events, 

the hash and tree are connected as well, serie now is maintaining a live in memory representation of processes
