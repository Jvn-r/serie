# DELI 14 - Logger Integ

## Objective
goal is to take the live in memory proc table and flush all the procs inside it to a json file, a more persistent way of storing procs
## Learnings
- so logger now, have to decide what to use to store data, the main options are json and csv
	- json : hierarchical, key value based, naturally maps to proc structs and variable length data
	- csv : simpler tables, widely used for ml and data analytics, could easily map to proc, but reduces visual relation ships, but doesnt naturally handle variable length fd arrays
- decided to use newline delimited json instead of regular JSON
    - normal JSON needs a single root object, so appending multiple logger flushes into one file isnt possible but NDJSON lets every process become one json object per line
- logger is intended for machine consumption, not humans
    - replay
    - ML datasets
    - analytics
    - future visualization
- decided not to store the process tree directly coz every proc already stores enough metadata to reconstruct the tree and as said before logs are for machine consimption
- possible future feature: rebuild the entire in memory process tree from a previous NDJSON log
- started using cJSON
	- single file dependency just add the `cJSON.c` and `cJSON.h` files and yore set, and has no external dependencies
- full logger pipeline is
```
proc table-> iterate every bucket-> iterate every proc-> proc -> cJSON object-> append json into one memory buffer-> single fputs into data.ndjson
```
-  learnt single large fputs is much faster than a lot of smaller fputs
- using one large write rather than lots of tiny writes is much faster
- built `logg_proc_2_json()`
    - maps almost the entire proc struct into json
    - integer metadata
    - names
    - lifecycle info
    - parent relationships
    - state
    - timestamps
    - alive state
    - exit code
    - later fd array will also be added
- realized `char` cannot be directly added into cJSON strings, had to convert process state into a temporary NULL terminated string
- changed ownership model of `logg_proc_2_json()`
    - initially it created and returned a cJSON object, but it was just wasting memory so later changed it to accept an already created object, reduces allocs and ownership becomes cleaner
- first tested the single process pipeline `logg_proc_2_json`
    - `proc *`into  `cJSON` obj printed json string using `cJSON_Print`, wrote into file
    - validated under valgrind
 - fork and exit time doesnt show like the clock time but rather a relative time to the dependent on the boot time
	 -   using `clock_gettime` and some math and `struct tm`  to resolve it 
    - still has a bug where resolved dates are incorrect, needs further investigation
- helper functions for math
    - `timespec_add()`
    - `timespec_sub()`
    - `logg_time_reso()`
- reading entire proc table
    - iterate all hash buckets
    - iterate every bucket chain
    - serialize every proc
    - append into one memory buffer
    - flush into `data.ndjson`
- MORE C pointer lessons
    - passing `char *` into the helper didnt work because realloc changes the pointer locally, so external fn didnt know abut the update so valgrind freaked out
    - changed helper to take `char **`, and caller now gets the updated allocation after realloc
- and operator precedence pain with brackets and pointers
    - `[]` binds before `*` so `*json_buff[index]` is not the same as `(*json_buff)[index]` and it was another gigantic valgrind freakout
    - fixed another segfault 
- spent a lot of time chasing down
    - realloc bugs
    - double frees, oh man
    - ownership mistakes
    - segfaults
    - memory leaks
    - valgrind carried the debugging process
- for now logger currently flushes on SIGINT
    - entire proc table is serialized into NDJSON
- first time i ran it i saw like 30 procs and thought i had hit some actual bottlenecks, but the logger runs after everythin soo WHAT, it was an `i++` at the end of my for loop, i forgit to remove it after i changed from a while loop, that was something
- and finally clean shutdown with zero memory leaks

## Outcome
live kernel events become in process objects, and the entire process table can now be put into a machine readable NDJSON log that will become ready for replay and analytics, still has some issues

current logger flow:
```text
Kernel Events
      |
      v
Process Table
      |
      v
Proc -> cJSON
      |
      v
Memory Buffer
      |
      v
data.ndjson
```
