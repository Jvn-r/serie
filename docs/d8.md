# DELI 8 - Architecture Shift planning

## Objective
- research about moving from a polling based to an event based system, using proc connector

## Learnings
- netlink : IPC mech bw kernel and userspace
- proc conn is built on top of netlink, and provides events from kernel like fork, exec, exit etc 
- why event driven
    - the kernel updates you whenever an event happens, so perfect timestamps and never miss anything
    - more complex, but you dont have to look, and reduces number of times the procfs is opened, which was the main time consumer in polling
- proc conn streams events only to processes that are subscribed to it 
- proc conn messages are wrapped in 2 outer layers
    - netlink msg 
    - connector msg 
    - proc event msg : where the actual data i need resides 
- proc conn is implemented by basically unwrapping these messages to get to the data 
`As I mentioned before, the proc connector is built on top of the generic connector and that itself is on top of netlink so sending that subscription message also involves embedded a message, inside a message inside a message. If you understood Christopher Nolan’s Inception, you should do just fine.` - https://nick-black.com/dankwiki/index.php/The_Proc_Connector_and_Socket_Filters
- i understood Inception, spinning top = dream
- now for things that arent Inception
- process storage = hash table, process relationships = proc tree 
- PID reuse has to be handled, could use pid + timestamps to mek them unique
- orphan process handled by revisiting /proc, as they are re-parented 
- data to be flushed into a log file of sorts to facilitate future analysis and a possible machine learning model
