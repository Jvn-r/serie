# DELI 13 - Mem Cleanups and Exit Integ

## Objective
- after getting fork working, a bunch of cleanup and lifecycle stuff surfaced that needed to be done before moving forward, then finally implemented `even_exit` and fixed a mem leak

## Learnings
- first order of business was proper cleanup, because valgrind showed a lot of "still reachable: 3,283,008 bytes in 821 blocks" from all the proc nodes being stored in the hash and tree
- implemented `SIGINT` signal interrupt
	- added a signal handler using `volatile sig_atomic_t running`
	- instead of killing the program immediately, Serie now
		- unregisters proc conn
		- closes the netlink socket
		- destroys the entire proc table
- added `proc_dele()`
	- removes a proc from the hash
	- uses both pid and birth timestamp to safely distinguish pid reuse
	- calls `proc_dest()` so there is only one owner responsible for freeing a proc
- validated cleanup with valgrind, all allocations were successfully reclaimed
- moved onto `even_exit()`
	- update alive = 0
	- record death_time
	- detach the dying proc from its parent
	- handle children
		- walk thru all children and set the children's parent(the exit proc) to NULL
		- reparent all children
			- theyre gonna be reparented by the kernel so we just have to ask the procfs to update the metadat for the orphan children
			- call atta_child in a loop with the new parent pid
	- keep parent inside the hash until we flush out for logging
- realised one `ppid` field wasnt enough
- `ppid`
	- rolling parent pid
	- refreshed from ProcFS whenever metadata is updated
	- represents the live kernel parent
- `birth_ppid`
	- captured only once
	- preserves the original creator process
	- never changes during reparenting
- dead processes intentionally remain inside the hash after exit
	- logger still needs the complete lifecycle
	- deletion will happen after logging / flushing
- extracted reparenting into `re_pare()`
	- keeps `even_exit()` readable
	- refreshes child metadata from proc fs
	- discovers the new parent from the kernel
	- reconstructs missing parents if needed
	- attaches orphan children back into the tree
- found an mem leak in valgrind report
	- `prfs_fd()` originally assumed it only populated new process objects
	- after introducing reparenting, `prfs_look()` could now refresh an already existing proc
	- this caused old fd snapshots to be overwritten with new allocations
	- fixed by cleaning existing fd allocations before rebuilding the snapshot
	- `prfs_look()` is now safe to call multiple times on the same proc
